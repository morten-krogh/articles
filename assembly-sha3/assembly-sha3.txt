An Assembly Implementation of SHA-3 Using AI

# Assembly coding is natural for AI

AI has revolutionized coding. In the last year or so, AI coding tools have
evolved from advanced code completion in editors into full-fledged autonomous
coding agents. In a future article, I will describe my general views and
predictions about AI in coding. Here, however, I will focus on the use of AI
for assembly coding.

# The SHA-3 hash algorithm as a test case

I have been curious for some time about whether AI is good at assembly coding,
so I decided to perform an experiment. I selected a well-defined and
interesting task that was not too large. My choice was the SHA3-256 hash
function. There were several reasons for that choice. First, I was familiar
with the principles and algorithm behind it. I had previously written a
JavaScript implementation by hand. Second, I appreciate its design principles.
Third, it is an important, well-known, and standardized algorithm.

# Summary of SHA-3

Keccak is the underlying algorithm family; SHA-3 is the NIST-standardized hash
family built from Keccak. NIST selected Keccak as the winner of the SHA-3
hash-function competition in 2012, and FIPS 202 defines the SHA-3 family as
a set of functions based on Keccak. This relationship is specified in
[FIPS 202](https://nvlpubs.nist.gov/nistpubs/fips/nist.fips.202.pdf).

The SHA3-256 hash function takes a sequence of bytes as input and returns 32
bytes, or 256 bits. The 256-bit output is conveniently represented as a
64-character hexadecimal string. For example:

sha3-256([30, 40]) = "bab9db9ee55722e9f6dda3716567417ff4084d972d4f0a9f32fe1b9f783b97ff"

More generally, SHA3-256 can take a bit string whose length need not be a
multiple of eight. We do not consider that case here and assume that the input
consists of bytes.

Briefly, the algorithm works as follows. It has a 200-byte, or 1,600-bit, state
that is initially filled with zeros. The input byte sequence is first padded in
a precise way so that it can be divided into an integral number of 136-byte
blocks. Each block is processed by first XORing it into the first 136 bytes of
the state and then permuting the bits in the state over 24 rounds. Each round,
called a Keccak round, consists of five steps. These steps perform both linear
and nonlinear operations on the 1,600 bits. The last of the five steps mixes a
fixed constant into the state, with a different constant used in each of the
24 rounds. The rounds are otherwise identical. The upshot is that the original
bytes are absorbed into the state, and the state itself is scrambled. After
all the blocks have been mixed in, the final hash is read from the first 32
bytes of the state.

This construction is called the sponge construction. It has two phases. The
first phase is the absorption phase, in which data is absorbed into the sponge
and scrambled. The second phase is the squeezing phase, in which bytes are
read from the sponge. In SHA3-256, the entire 32-byte digest fits within a
single output block. There are other uses of the Keccak sponge, such as the
SHAKE functions, in which the sponge can undergo the 24 Keccak rounds
repeatedly to produce multiple blocks of output.

The sponge construction is very elegant and can be used for purposes other
than the SHA3-256 hash function, but that is beyond the scope of this article.

# How the coding agents were used

I "collaborated" with several coding agents: Codex using GPT-5.5 and GPT-5.6,
and Claude using Opus 4.8 and Fable 5. The point was not to compare the agents
with one another, but rather to work with them as a team. The code was written
entirely by the agents. My role was to define and refine the goals, write
prompts, occasionally inspect the code, suggest improvements such as reducing
buffer copies, inlining functions, and changing function signatures, and
repeatedly ask the agents to look for further improvements. Performance and
code size were measured regularly. The goal was to maximize performance
without accepting tiny gains that required a much larger code size.

# Implementations

The implementations, together with others, are published in this
[GitHub repository](https://github.com/morten-krogh/algorithms/tree/main). The
repository contains the assembly files together with some tests, benchmarks,
and command-line invocations of the hash functions.

There are three implementations of the SHA3-256 algorithm: one in WebAssembly,
one in x86-64 assembly for Linux, and one in Arm64 assembly for macOS. The core
code is written directly in each platform's assembly format: WebAssembly Text
(`.wat`) for WebAssembly, x86-64 assembly for Linux, and Arm64 assembly for
macOS.

Below are three benchmark tables. The rows represent different input sizes.
Each table compares one of these implementations with two alternatives
appropriate to that platform. In general, the efficiency of the Keccak rounds
matters more for larger inputs. For smaller inputs, the benchmarks are
dominated by copying and crossing language barriers. The larger inputs are
probably the most interesting cases.

"sha3-256-wasm" is the implementation presented here. The
"hash-wasm" implementation is the highest-performance published
JavaScript/WebAssembly implementation I could find. The "node" implementation
runs Node.js with OpenSSL, so it uses native code on the M1 Mac where this
benchmark was run. We see that the new implementation is faster than the other
WebAssembly implementation. For large inputs, it delivers about 30% less
throughput than the native Node.js/OpenSSL implementation. For small inputs,
the current "sha3-256-wasm" implementation is actually faster than the
Node.js/OpenSSL implementation. In general, performance for small inputs is
mostly determined by buffer copying and crossing language barriers. For large
inputs, performance depends mostly on the efficiency of the Keccak rounds,
although it can always be negatively affected by unnecessary copies. V8
initially compiles WebAssembly with a fast baseline compiler, then optimizes
hot code later. To avoid measuring that warm-up period, all implementations
were warmed up before benchmarking.

The "sha3-256-x86_64-linux" implementation is the one presented here. It is
compared with OpenSSL and gcrypt. We see that the current implementation is
consistently faster than OpenSSL and on par with gcrypt. OpenSSL therefore
clearly leaves some performance on the table. We believe that the current
implementation and the gcrypt implementation are close to the maximum
performance achievable on the tested system. At the three largest input sizes,
the current implementation is about 0.5%-0.9% faster. A likely explanation is
that it is hard-coded for SHA3-256 rather than being a generic SHA-3
implementation that supports other output sizes, such as SHA3-512. The
difference is not important, but it illustrates an important point: in an
AI-assisted programming environment, the need for generic code is lower than
in a traditional programming environment, where we generally prefer code
reuse.

# Conclusions

I would say that this coding experiment validated the hypothesis that AI agents
can write assembly code directly. They do not need to express a solution in a
higher-level language and then rely on a compiler to generate assembly. They
can optimize the instructions in an assembly program for performance and code
size. Also, hard-coding the exact parameters for SHA3-256 rather than
implementing generic support for variants such as SHA3-512 can,
unsurprisingly, lead to higher performance and smaller code sizes. What is new
here is that pursuing such optimizations is feasible because AI-assisted
programming is so fast relative to human programming.

I will leave it at that here and elaborate on my thoughts on programming
in future articles.
