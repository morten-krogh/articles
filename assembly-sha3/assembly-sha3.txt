# Summary of Sha3

Keccak is the underlying algorithm family; SHA-3 is the NIST-standardized hash family built from Keccak.
NIST selected Keccak as the winner of the SHA-3 hash-function competition in 2012. , 

and FIPS 202 defines the SHA-3 family as functions based on instances of Keccak

https://nvlpubs.nist.gov/nistpubs/fips/nist.fips.202.pdf

The hash function takes an input
which is a sequence of bytes and returns a sequence of 32 bytes = 256 bits. The output is
conveniently represented as a hex string of length 64.
For example,

sha3-256([30, 40]) = "bab9db9ee55722e9f6dda3716567417ff4084d972d4f0a9f32fe1b9f783b97ff"

Actually, the input could be a sequence of bits that do not make up an integer number of bytes,
but we do not care about that here, and assume that the input consists of bytes.

Briefly, the algoritmh works like this. There is a 200 byte = 1600 bits state
that starts out as zero filled. The input byte sequence is first padded in a
precise way such that the input can be broken into an integral number of blocks
of 136 bytes. Each block is processed by first xor'ing it into the first 136
bytes of the state and then permuting the bits in the state in 24 rounds. Each
round, called a Keccak round, consists of 5 steps itself. The steps consists of
both linear and non-linear operations of the 1600 bits. Furthermore, there is
also a step that mixes completely fixed constants into the state. There is one
fixed constant for each of the 24 round. The 24 rounds are identical except for
the constants used. The upshot is that the original bytes are absorbed into the state and the state 
itself is scrambled. After mixing all blocks in, the final result of the hash is read out from the 
first 32 bytes of the state. 

This construction is called the sponge construction. It has two phases. The
first phase is the absorption phase where data is absorbed into the sponge and
scrambled. The second phase is the squeezing phase, where bytes are read from
the sponge. In the case of the sha3-256, the readout is short, but if the
readout had been longer, the sponge would have undergone the 24 Keccak rounds
repeatedly for each block of output.

The sponge construction is very elegant and can be used for other purposes than the sha3-256 
hash function, but that is beyond this article.

# How the coding agents were used

The procedure was to collaborate with the coding agents, Codex with GPT5.5 and
GPT5.6 and Claude with Opus4.8 and Fable 5. The point was not to compare the
agents with each other, but rather a team work between me and them. The code
was written purely by the agents. My role was to drive the goals, write
prompts, update the goals, occasionally look a the code, suggest some
improvements such as less copying of buffers, change the API, or to ask them to
look again for improvements. The performance and cose size were measured
regularly. The goal was to have as high performance as possible, except that
tiny performance improvements at the expense of a much larger code size were
skipped.
